What is Containerization?

Containerization packages an application and all its dependencies — runtime, libraries, configuration — into a single, portable unit called a container. Unlike a virtual machine (VM), containers don't carry a full OS; they share the host OS kernel, making them lightweight and fast.

The problem containers solve is the infamous "it works on my machine" issue. By bundling the exact environment an application needs, containers ensure consistent behaviour across development, staging, and production.

🚢 Analogy
Think of shipping containers. Before standardized containers, every ship had to be loaded cargo piece by piece — slow and unpredictable. The standardized container changed everything: any cargo (your app), in the same box (Docker container), fits on any ship (any host). Ports (servers) don't care what's inside; they just move containers.
VM vs. Container Architecture
📦
Docker Core Concepts

A Docker image is a read-only template with instructions for creating a container. Think of it like a class in object-oriented programming — the image is the blueprint, and containers are the instances. Images are built from a Dockerfile using the docker build command.

Dockerfile — building a Node.js app image
# Base image — start from official Node 20 alpine
FROM node:20-alpine

# Set working directory inside the container
WORKDIR /app

# Copy dependency files first (layer caching benefit)
COPY package*.json ./
RUN npm ci --only=production

# Copy the rest of the application source
COPY . .

# Expose port the app listens on
EXPOSE 3000

# Non-root user for security
USER node

# Default command to run
CMD ["node", "server.js"]

Build this with: docker build -t my-app:v1 . — the -t flag tags the image with a name and version.

A container is a running instance of an image. Containers are ephemeral by default — when they stop, any data written inside is lost unless you mount a volume. Multiple containers can run from the same image simultaneously.

Essential docker container commands
# Run a container (detached, port mapping)
docker run -d -p 3000:3000 --name myapp my-app:v1

# List running containers
docker ps

# View logs (follow mode)
docker logs -f myapp

# Execute a command inside a running container
docker exec -it myapp sh

# Stop and remove
docker stop myapp && docker rm myapp

# Run with a volume for persistent storage
docker run -v /host/data:/app/data my-app:v1

A registry is a storage and distribution system for Docker images. Docker Hub is the default public registry. Companies use private registries (AWS ECR, Google Artifact Registry, GitHub Container Registry, Harbor) to store proprietary images.

Push/pull images from registries
# Tag for Docker Hub (user/image:tag)
docker tag my-app:v1 username/my-app:v1

# Login and push
docker login
docker push username/my-app:v1

# Pull from AWS ECR (private registry)
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin \
  123456789.dkr.ecr.us-east-1.amazonaws.com

docker pull 123456789.dkr.ecr.us-east-1.amazonaws.com/my-app:v1

Docker images are built in layers. Each instruction in a Dockerfile (FROM, RUN, COPY, etc.) creates a new layer. Layers are cached — if a layer hasn't changed, Docker reuses the cached version, massively speeding up builds.

💡 Layer Caching Strategy

Always put frequently-changing instructions (copying app code) after rarely-changing ones (installing dependencies). This ensures only the changed layers rebuild. In our Dockerfile example, COPY package*.json comes before COPY . . precisely for this reason.

Docker Image Layer Stack

Docker provides several network drivers. bridge (default) creates an isolated network; containers can communicate by container name. host shares the host's network stack. overlay enables multi-host networking (used in Swarm/Kubernetes). none disables networking entirely.

Docker networking basics
# Create a custom bridge network
docker network create my-net

# Run containers on the same network
docker run -d --network my-net --name db postgres:15
docker run -d --network my-net --name api my-app:v1

# Now api can reach db via hostname "db"
# DB_HOST=db works inside the api container
docker exec api ping db
🔗
Docker Compose: Multi-Container Apps

Real applications rarely run as a single container. A web app typically needs a database, cache, background worker, and maybe a reverse proxy. Docker Compose lets you define all these services in a single docker-compose.yml file and manage them with one command.

docker-compose.yml — Full-stack web application
version: '3.9'

services:
  # PostgreSQL database
  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: app_user
      POSTGRES_PASSWORD: ${DB_PASSWORD}   # from .env
    volumes:
      - pg_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app_user"]
      interval: 10s
      timeout: 5s
      retries: 5

  # Redis cache
  redis:
    image: redis:7-alpine
    command: redis-server --requirepass ${REDIS_PASS}

  # Node.js API — built from local Dockerfile
  api:
    build: .
    ports:
      - "3000:3000"
    environment:
      DB_HOST: db
      REDIS_HOST: redis
    depends_on:
      db:
        condition: service_healthy
    volumes:
      - ./src:/app/src   # dev hot-reload

volumes:
  pg_data:

Run the whole stack with docker compose up -d. Tear it all down with docker compose down -v (the -v also removes volumes).

⚠️ Compose vs. Production

Docker Compose is excellent for local development and simple single-host setups. For production at scale — multiple hosts, rolling updates, self-healing — you need Kubernetes (Module 2).

⚠️
Common Misconceptions
❌
"Containers are just lightweight VMs"
Reality: Containers share the host OS kernel and are fundamentally different from VMs. VMs virtualize hardware; containers virtualize the OS user space. A container can start in milliseconds; a VM takes minutes.
❌
"Docker containers are stateless by definition"
Reality: Containers are ephemeral by default, but you can persist data using volumes. The container itself doesn't store state permanently — the volume does.
❌
"Running as root inside a container is fine since it's isolated"
Reality: Container isolation is not perfect. Running as root creates security risks — if a container is compromised, an attacker may escape to the host. Always use the USER directive with a non-root user.
PRACTICE
Containerize a Python Flask App
  1. Create a simple app.py with Flask that returns a JSON health response at /health
  2. Write a Dockerfile using python:3.11-slim as base. Install dependencies with pip install --no-cache-dir
  3. Add a .dockerignore file excluding __pycache__, .env, and *.pyc
  4. Build the image: docker build -t flask-app:dev . and verify with docker images
  5. Run the container and test: curl http://localhost:5000/health
  6. Add a Redis service in docker-compose.yml and update your app to use it as a request counter
  7. Compare the image sizes — run docker history flask-app:dev to see the layers
🎯
Key Takeaways
Images are blueprintsContainers run from images; images are immutable and built in layers
Layers mean cachingOrder your Dockerfile instructions from least to most frequently changing
Ephemeral by defaultUse named volumes for any data that must outlive the container lifecycle
Security mattersNever run containers as root; use minimal base images like alpine or slim
Compose for devDocker Compose orchestrates multi-container setups on a single host
Registry is the hubPush/pull images from registries to share across teams and environments

Why Kubernetes?

When you run dozens or hundreds of containers across multiple servers, Docker Compose breaks down. You need a system that handles scheduling (which container runs on which host), self-healing (restart crashed containers), scaling (add/remove replicas), rolling updates, and service discovery.

Kubernetes (K8s) is the de-facto standard container orchestration platform. Originally designed by Google (based on their internal Borg system), it was open-sourced in 2014 and is now maintained by the CNCF.

✈️ Analogy
If Docker containers are individual flights, Kubernetes is the air traffic control system. It knows about every runway (node), every plane (container), tracks capacity, routes traffic to the right destination, and automatically re-routes when a runway closes (node failure) — without passengers (users) ever noticing.
Kubernetes Architecture
🧩
Core Kubernetes Objects

A Pod is the smallest deployable unit in Kubernetes. A pod wraps one or more containers that share the same network namespace (same IP) and can share volumes. Typically a pod runs a single container; sidecar containers (log shippers, service meshes) are the main reason for multi-container pods.

pod.yaml — minimal pod definition
apiVersion: v1
kind: Pod
metadata:
  name: api-pod
  labels:
    app: api
spec:
  containers:
    - name: api
      image: my-app:v1
      ports:
        - containerPort: 3000
      resources:
        requests:
          memory: "64Mi"
          cpu: "250m"
        limits:
          memory: "128Mi"
          cpu: "500m"
⚠️ Don't run raw pods in production

Raw pods aren't rescheduled if a node fails. Always use a Deployment which manages pod replicas and handles failures automatically.

A Deployment describes the desired state of a set of pods: which image to run, how many replicas, update strategy. The Deployment Controller constantly reconciles actual state with desired state.

deployment.yaml — production deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1       # extra pods during update
      maxUnavailable: 0  # zero downtime
  template:
    metadata:
      labels:
        app: api
    spec:
      containers:
        - name: api
          image: my-app:v2   # update image tag to roll out
          readinessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 5
          livenessProbe:
            httpGet:
              path: /health
              port: 3000
            periodSeconds: 15

Pods are ephemeral — they get new IPs every time they restart. A Service provides a stable IP and DNS name that routes traffic to matching pods (via label selectors). Types: ClusterIP (internal), NodePort (external on a static port), LoadBalancer (cloud LB), ExternalName (DNS alias).

service.yaml — ClusterIP + LoadBalancer
# Internal service (ClusterIP, default)
apiVersion: v1
kind: Service
metadata:
  name: api-service
spec:
  selector:
    app: api       # routes to pods with this label
  ports:
    - protocol: TCP
      port: 80          # service port
      targetPort: 3000  # container port
  type: ClusterIP

---
# External-facing LoadBalancer service
apiVersion: v1
kind: Service
metadata:
  name: api-lb
spec:
  type: LoadBalancer
  selector:
    app: api
  ports:
    - port: 443
      targetPort: 3000

ConfigMap stores non-sensitive configuration as key-value pairs, decoupling config from image. Secret stores sensitive data (passwords, tokens) encoded as base64 — mounted as env vars or files. Always use sealed-secrets or external secret managers (Vault, AWS Secrets Manager) in production.

configmap and secret
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DB_HOST: "postgres-svc"
  LOG_LEVEL: "info"
  MAX_CONNECTIONS: "100"
---
apiVersion: v1
kind: Secret
metadata:
  name: app-secrets
type: Opaque
data:                          # base64 encoded
  DB_PASSWORD: cGFzc3dvcmQxMjM=
  JWT_SECRET: c3VwZXJzZWNyZXQ=

# Reference in deployment:
envFrom:
  - configMapRef:
      name: app-config
  - secretRef:
      name: app-secrets

For stateful apps, Kubernetes provides PersistentVolumes (PV) — a piece of storage provisioned in the cluster. A PersistentVolumeClaim (PVC) is a request for storage by a pod. StorageClasses enable dynamic provisioning (auto-create cloud disks on demand).

PVC for stateful database storage
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-pvc
spec:
  accessModes:
    - ReadWriteOnce     # single node read-write
  storageClassName: gp3  # AWS EBS gp3
  resources:
    requests:
      storage: 20Gi

An Ingress manages external HTTP/HTTPS access to services — hostname-based routing, path-based routing, TLS termination. It requires an Ingress Controller (NGINX, Traefik, AWS ALB) to be installed in the cluster.

ingress.yaml — path-based routing with TLS
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: main-ingress
  annotations:
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
spec:
  tls:
    - hosts:
        - api.example.com
      secretName: api-tls-cert
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: api-service
                port:
                  number: 80
⛵
Helm: The Kubernetes Package Manager

Managing dozens of Kubernetes YAML files becomes unwieldy. Helm is the package manager for Kubernetes — it bundles related K8s manifests into a chart, lets you template values (e.g., different replicas for dev vs prod), and tracks releases.

Helm cheat sheet
# Install a chart from Helm Hub
helm install my-nginx ingress-nginx/ingress-nginx

# Install with custom values file
helm install my-app ./charts/my-app -f values-prod.yaml

# Upgrade a release
helm upgrade my-app ./charts/my-app --set image.tag=v2.0

# List all releases
helm list --all-namespaces

# Rollback to previous version
helm rollback my-app 1

# Render templates without installing (dry-run)
helm template my-app ./charts/my-app
⚠️
Common Misconceptions
❌
"Kubernetes is only for large companies"
Reality: K8s is used from solo projects to Fortune 500 companies. Managed services like EKS, GKE, and AKS make the operational overhead minimal. Even a small team benefits from self-healing and declarative deployments.
❌
"I can store sensitive data in ConfigMaps"
Reality: ConfigMaps are not encrypted. Never store passwords, API keys, or tokens in ConfigMaps. Use Secrets (and encrypt etcd at rest) or better — integrate with a dedicated secret manager like HashiCorp Vault.
PRACTICE
Deploy a Scalable Web App on Kubernetes
  1. Install minikube and kubectl. Start a local cluster: minikube start --nodes 2
  2. Write a Deployment YAML for your Docker image with 3 replicas, resource limits, and health probes
  3. Create a ClusterIP Service pointing to your Deployment
  4. Store your DB password in a K8s Secret and reference it in your Deployment
  5. Add a Horizontal Pod Autoscaler: kubectl autoscale deployment api --min=2 --max=10 --cpu-percent=60
  6. Simulate load with kubectl run -it --rm load --image=busybox -- /bin/sh -c "while true; do wget -q api-svc; done" and watch pods scale
  7. Perform a rolling update by changing the image tag and run kubectl rollout status deployment/api
🎯
Key Takeaways
Desired state modelTell K8s what you want; it figures out how to get there and stays there
Labels tie everythingServices find pods via label selectors — labels are the fundamental glue
Readiness vs LivenessReadiness gates traffic; Liveness restarts unhealthy pods — both matter
Set resource limitsWithout CPU/memory limits one noisy pod can starve an entire node
Helm for real appsUse Helm for anything beyond toy deployments; it enables versioned releases
Namespaces for isolationSeparate environments (dev/staging/prod) into different K8s namespaces

CI/CD: Automating the Path to Production

Continuous Integration (CI) is the practice of automatically building and testing code every time a developer pushes a change. Continuous Delivery/Deployment (CD) extends this — automatically deploying tested code to staging or production environments.

A CI/CD pipeline is a sequence of automated stages: code commit → build → test → security scan → package → deploy → verify.

🏭 Analogy
CI/CD is like a factory assembly line. Raw material (code) enters one end. At each station (stage), something is added or checked. Quality control (tests, scans) happens automatically at every step. Only products (builds) that pass every inspection reach the store shelves (production). A rejected part stops the line early, not after reaching customers.
CI/CD Pipeline — From commit to production
⚙️
GitHub Actions: Pipelines as Code

GitHub Actions is one of the most popular CI/CD platforms. Pipelines are defined as YAML files in .github/workflows/. They are triggered by GitHub events (push, PR, schedule, manual) and run in virtual runners (Ubuntu, Windows, macOS).

.github/workflows/deploy.yml — Full CI/CD pipeline
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  IMAGE_NAME: myapp
  REGISTRY: ghcr.io/myorg

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm test
      - run: npm run lint

  security-scan:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Trivy vulnerability scan
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'
          severity: 'CRITICAL,HIGH'
          exit-code: '1'

  build-and-push:
    needs: security-scan
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - name: Login to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to Kubernetes
        run: |
          kubectl set image deployment/api \
            api=${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
          kubectl rollout status deployment/api

Terraform: Infrastructure as Code

Infrastructure as Code (IaC) means managing infrastructure (servers, networks, databases, DNS) through machine-readable configuration files rather than manual UI clicks. This enables version control, repeatability, and collaborative review of infrastructure changes.

Terraform by HashiCorp is the most widely-used IaC tool. It uses HCL (HashiCorp Configuration Language) and supports 1000+ providers (AWS, GCP, Azure, Kubernetes, Datadog, GitHub, etc.).

main.tf — Provision an EKS cluster on AWS
# Configure AWS provider
provider "aws" {
  region = "us-east-1"
}

# Remote state in S3 (team collaboration)
terraform {
  backend "s3" {
    bucket = "my-tf-state"
    key    = "prod/terraform.tfstate"
    region = "us-east-1"
    dynamodb_table = "tf-lock"   # state locking
  }
}

# VPC for our cluster
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.0.0"
  name    = "prod-vpc"
  cidr    = "10.0.0.0/16"
  azs     = ["us-east-1a", "us-east-1b", "us-east-1c"]
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24", "10.0.103.0/24"]
  enable_nat_gateway = true
}

# EKS Cluster
module "eks" {
  source  = "terraform-aws-modules/eks/aws"
  version = "19.0.0"

  cluster_name    = "prod-cluster"
  cluster_version = "1.28"
  vpc_id     = module.vpc.vpc_id
  subnet_ids = module.vpc.private_subnets

  eks_managed_node_groups = {
    workers = {
      instance_types = ["t3.medium"]
      min_size = 2; max_size = 10; desired_size = 3
    }
  }
}
Terraform Workflow
terraform init → download providers
terraform plan → preview changes
terraform apply → create resources
terraform destroy → tear down
State Management
Terraform tracks resources in a state file (.tfstate). Always store state remotely (S3 + DynamoDB lock) in team environments — never commit it to Git.
🔀
GitOps: Git as the Single Source of Truth

GitOps extends IaC to deployments: the desired state of your system lives entirely in Git. A GitOps agent (ArgoCD, Flux) watches the Git repo and reconciles the cluster to match what's committed. Any change to production goes through a PR — giving you full audit history, easy rollbacks, and clear ownership.

Aspect Traditional CI/CD GitOps
Deployment trigger CI pipeline pushes directly Git commit (declarative)
Rollback Re-run pipeline with old tag git revert → auto-synced
Drift detection Manual or none Continuous reconciliation
Audit trail CI logs (ephemeral) Full Git history
Tool examples Jenkins, GitHub Actions ArgoCD, FluxCD
⚠️
Common Misconceptions
❌
"Terraform state file can be committed to Git"
Reality: The state file often contains sensitive data (passwords, keys) in plaintext. Always store remotely (S3, Terraform Cloud). Also never delete it — Terraform loses track of what it manages.
❌
"CI/CD means everything deploys automatically to production"
Reality: Continuous Delivery means you can deploy at any time; the actual deploy may require manual approval. Continuous Deployment means it does deploy automatically. Most teams use Delivery for production and Deployment for staging.
PRACTICE
Build a Full CI/CD Pipeline with GitHub Actions
  1. Create a GitHub repo with a simple Node.js app. Add Jest tests that you will make pass and fail to see the pipeline react
  2. Create .github/workflows/ci.yml that runs on every push and PR — install, lint, test
  3. Add a Docker build step that pushes to GHCR only on main branch merges
  4. Add a secret KUBECONFIG in GitHub secrets and add a deploy step that runs kubectl apply
  5. Write a main.tf that creates an S3 bucket with versioning enabled. Run terraform plan and review output
  6. Break a test intentionally and push. Observe the pipeline fail and block deployment. Fix and see it succeed
🎯
Key Takeaways
Fail fastCatch problems in CI (lint, tests, security) before they ever reach production
Immutable artifactsBuild Docker images once; promote the same artifact through envs
Secrets in CIUse environment secrets in GitHub/GitLab; never hardcode in workflow YAML
Terraform is declarativeDescribe infrastructure, not procedures — Terraform figures out the order
GitOps = auditabilityEvery production change traceable to a PR, reviewer, and commit hash
Remote state is a mustS3 + DynamoDB locking prevents state corruption in team workflows

Observability: The Three Pillars

In distributed systems, you can't manually inspect every machine. Observability is the ability to understand a system's internal state from its external outputs. There are three pillars — and you need all three to effectively debug production issues.

📈
Metrics
Aggregated numerical measurements over time. CPU %, request rate, error rate, latency. Cheap to store, great for dashboards and alerting. Tool: Prometheus + Grafana
📝
Logs
Timestamped records of discrete events. Your best friend for root cause analysis. Expensive at scale — need good aggregation. Tool: ELK Stack, Loki, Datadog
🔍
Traces
End-to-end request path across microservices. Shows exactly where latency or errors occur. Tool: Jaeger, Zipkin, Tempo
🏥 Analogy
A hospital patient monitoring system: metrics are the vitals (heart rate, blood pressure — numbers over time), logs are the doctor's notes (discrete events with context), and traces are the patient's journey through different departments (path through the system). You need all three to diagnose what went wrong and when.
🔥
Prometheus & Grafana: The Monitoring Stack

Prometheus is a time-series database that scrapes metrics from your services. Grafana visualizes those metrics on dashboards and fires alerts. Together, they're the most popular open-source monitoring stack.

prometheus.yml — scrape configuration
global:
  scrape_interval: 15s
  evaluation_interval: 15s

rule_files:
  - "alerts/*.yml"

alerting:
  alertmanagers:
    - static_configs:
        - targets: ["alertmanager:9093"]

scrape_configs:
  # Prometheus self-monitoring
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  # Kubernetes pods with annotation-based discovery
  - job_name: 'kubernetes-pods'
    kubernetes_sd_configs:

        expr: |
          histogram_quantile(0.99,
            rate(http_request_duration_seconds_bucket[5m])

        for: 2m

          severity: warning

      # Alert if pod count drops below desired
      - alert: PodNotReady
        expr: |
          kube_deployment_status_replicas_ready
          < kube_deployment_spec_replicas
        for: 2m
        labels:

Applications must expose metrics for Prometheus to scrape. The standard is a /metrics HTTP endpoint in Prometheus exposition format. Use client libraries for your language.

Node.js — instrumenting with prom-client
const promClient = require('prom-client');

// Collect default Node.js metrics (event loop, gc, etc.)
promClient.collectDefaultMetrics();

// Custom HTTP request counter
const httpRequests = new promClient.Counter({
  name: 'http_requests_total',
  help: 'Total HTTP requests',
  labelNames: ['method', 'route', 'status']
});

// Custom histogram for latency
const httpDuration = new promClient.Histogram({
  name: 'http_request_duration_seconds',
  help: 'HTTP request latency',
  labelNames: ['route'],
  buckets: [0.1, 0.3, 0.5, 1, 2, 5]
});

// Express middleware
app.use((req, res, next) => {
  const end = httpDuration.startTimer({ route: req.path });
  res.on('finish', () => {
    httpRequests.inc({ method: req.method, route: req.path, status: res.statusCode });
    end();
  });
  next();
});

// Metrics endpoint for Prometheus to scrape
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', promClient.register.contentType);
  res.end(await promClient.register.metrics());
});

OpenTelemetry (OTel) is the emerging standard for instrumentation — a vendor-neutral SDK for collecting metrics, logs, and traces in one unified framework. You instrument once and send data to any backend (Jaeger, Grafana Tempo, Datadog, etc.).

OpenTelemetry setup for Node.js
const { NodeSDK } = require('@opentelemetry/sdk-node');
const { OTLPTraceExporter } = require('@opentelemetry/exporter-trace-otlp-http');
const { getNodeAutoInstrumentations } = require('@opentelemetry/auto-instrumentations-node');

const sdk = new NodeSDK({
  traceExporter: new OTLPTraceExporter({
    url: 'http://otel-collector:4318/v1/traces'
  }),
  instrumentations: [getNodeAutoInstrumentations()]  // auto instruments HTTP, Express, DB
});

sdk.start();

// Your app is now automatically traced — every HTTP request
// creates spans with timing, status codes, and propagated context

SLIs, SLOs, and Error Budgets

SLI (Service Level Indicator): a specific metric that measures service health. E.g., "the fraction of successful HTTP requests."

SLO (Service Level Objective): the target value for an SLI. E.g., "99.9% of requests succeed". This is your reliability goal.

Error Budget: the allowed failure rate = 100% - SLO. A 99.9% SLO gives you 0.1% to "spend" on outages — ~43.8 minutes per month. When the budget is burned, the team freezes new feature work to focus on reliability.

🔍 Why error budgets work

Error budgets remove the "dev vs ops" tension. Developers want to ship features; SREs want stability. The error budget is a shared number that both teams own — it makes reliability a measurable business goal, not a vague preference.

DevSecOps: Security Baked In

DevSecOps integrates security practices directly into the DevOps pipeline — "shift left" means finding and fixing vulnerabilities early (at code time), not late (at deploy time or after a breach).

Common Misconceptions
❌
"Monitoring and observability are the same thing"
Reality: Monitoring tells you when something is wrong. Observability tells you why. Monitoring is about known unknowns (pre-defined alerts); observability lets you debug unknown unknowns (novel failures) using logs, traces, and metrics together.
❌
"Security can be added at the end of the DevOps cycle"
Reality: Security vulnerabilities are 30x cheaper to fix during development than after deployment. Shifting security left (into CI/CD) catches issues before they reach production and eliminates security as a deployment blocker.
